# VK Integration

## Короткий вывод

Для автопостинга текста в VK в ряде случаев может хватить `group/community token`, но для автопостинга поста с изображением проекту нужен **пользовательский VK access token** пользователя, который имеет права на нужное сообщество.

Если использовать `ключ сообщества` из раздела VK API, приложение обычно сможет:

- проверить существование группы;
- иногда опубликовать текст;
- но не сможет загрузить `wall photo`.

Именно поэтому в проекте может появляться сценарий: текст в VK опубликовался, а изображение было пропущено.

## Какие бывают VK-ключи

| Тип ключа | Где обычно получают | Для чего подходит | Ограничения в этом проекте |
|-----------|----------------------|-------------------|----------------------------|
| `group/community token` | Настройки сообщества -> API -> ключи доступа | Некоторые server-side операции от имени сообщества | Не подходит для `photos.getWallUploadServer`, поэтому пост с картинкой не проходит полностью |
| `user access token` | OAuth-авторизация пользователя через VK ID / VK Dev | Публикация на стену и сценарии, где VK требует user auth | Пользователь должен иметь реальные права в сообществе |

## Какой ключ нужен именно этому проекту

Для стабильного сценария `сгенерировать пост -> прикрепить изображение -> опубликовать в VK` нужен:

1. `user access token`
2. scope `wall`
3. scope `photos`
4. пользователь, который является администратором, владельцем или редактором сообщества

Проект хранит этот токен в пользовательских настройках как `vk_api_key`, но по смыслу это именно **VK user access token**.

## Почему одного scope недостаточно

VK проверяет не только список запрошенных прав, но и тип авторизации.

На практике это означает:

1. токен может содержать `wall, photos`;
2. `groups.getById` может успешно работать;
3. но `photos.getWallUploadServer(group_id=...)` все равно может падать.

Именно это произошло в текущем проекте. Повторная проверка показала:

- `groups.getById=OK`
- `photos.getWallUploadServer=API_ERROR`
- ответ VK: `[27] Group authorization failed: method is unavailable with group auth.`

Этот ответ означает, что VK видит **group auth**, а не user auth.

## Как проект валидирует VK-настройки

После сохранения настроек на странице `Settings` приложение делает две независимые проверки:

1. Проверяет доступ к группе через `groups.getById`.
2. Проверяет возможность загрузки wall photo через `photos.getWallUploadServer`.

Результат сохраняется в БД как capability metadata:

- `validation_status`
- `validation_message`
- `can_access_group`
- `can_post_to_wall`
- `can_upload_wall_photo`

Это нужно для двух целей:

1. показать пользователю понятное состояние интеграции;
2. не ломать весь workflow генерации, если VK поддерживает только часть операций.

## Как приложение ведет себя при разных VK-сценариях

| Сценарий | Что происходит |
|----------|----------------|
| Нет VK settings | Генерация проходит, автопостинг пропускается |
| Нет доступа к группе | Генерация проходит, автопостинг пропускается |
| Есть доступ к группе, но нет wall photo upload | Генерация проходит, пост публикуется в VK без изображения |
| Есть доступ и к группе, и к wall photo upload | Генерация проходит, пост публикуется в VK с изображением |
| VK отклоняет публикацию целиком | Генерация сохраняется, пользователь получает warning |

## Что нужно проверить в VK перед использованием токена

### 1. Тип токена

Нужен именно **user token**, а не `ключ сообщества`.

Неправильный источник:

- раздел сообщества `Работа с API` / `Ключи доступа`

Правильный источник:

- OAuth-авторизация пользователя через VK Dev / VK ID

### 2. Права токена

У user token должны быть:

- `wall`
- `photos`

### 3. Права пользователя в сообществе

Пользователь, от имени которого получен token, должен иметь право:

- публиковать на стену;
- работать с контентом сообщества;
- при необходимости публиковать от имени сообщества.

### 4. Настройки сообщества

Проверьте:

- стена сообщества не ограничена так, что публикация невозможна;
- у администратора/редактора сообщества действительно есть права на публикацию;
- используется правильный `group_id` именно сообщества, в которое идет публикация.

## Что именно нужно вводить в UI проекта

На странице `Settings`:

- в поле `vk_api_key` вставляется **user access token**;
- в поле `vk_group_id` вставляется числовой ID сообщества без `club` и без URL.

Пример:

- URL сообщества: `https://vk.com/club236361359`
- Значение для поля `vk_group_id`: `236361359`

## Симптомы и их смысл

| Что видит пользователь | Что это обычно значит |
|------------------------|------------------------|
| `Не удалось получить информацию о группе VK` | токен неверный, группа неверная или нет доступа к группе |
| `Группа доступна, но право загрузки wall photo недоступно` | чаще всего используется group token вместо user token |
| Текст постится, а изображение нет | есть частичная совместимость, но VK режет `wall photo upload` |
| `VK не принял публикацию или token не имеет нужных прав` | либо нет нужных scope, либо недостаточно реальных прав в сообществе |

## Что нужно исправлять в проекте, а что нет

### Не проблема проекта

Если VK отвечает:

`[27] Group authorization failed: method is unavailable with group auth.`

то это не баг OpenAI, не проблема Docker и не ошибка генерации изображения. Это означает, что токен неподходящего типа для загрузки фото на стену.

### Что проект уже делает правильно

Проект намеренно использует best-effort поведение:

1. не теряет уже сгенерированный контент;
2. не обрушает весь flow при отказе VK;
3. публикует текст без изображения, если именно загрузка wall photo недоступна.

### Что можно улучшить дальше

Если понадобится доработка UX, логично:

1. переименовать поле `vk_api_key` в `VK user access token`;
2. показывать в UI явный признак `group token detected`;
3. добавить короткую ссылку-инструкцию по получению user token.

## Практический чеклист

Перед повторной проверкой VK в проекте:

1. Получите новый `user access token`.
2. Убедитесь, что у него есть `wall` и `photos`.
3. Убедитесь, что этот пользователь управляет нужным сообществом.
4. Вставьте token в `Settings`.
5. Сохраните настройки.
6. Повторите публикацию с включенной генерацией изображения.

Ожидаемый результат после исправления:

1. warning про `wall photo upload` исчезает;
2. `can_upload_wall_photo` становится `true`;
3. пост уходит в VK уже вместе с изображением.
